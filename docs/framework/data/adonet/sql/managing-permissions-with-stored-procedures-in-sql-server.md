<span data-ttu-id="aad5a-104">데이터베이스 주변을 방어하는 여러 줄을 작성하는 한 가지 방법은 저장 프로시저 또는 사용자 정의 함수를 사용하여 모든 데이터 액세스를 구현하는 것입니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-104">One method of creating multiple lines of defense around your database is to implement all data access using stored procedures or user-defined functions.</span></span> <span data-ttu-id="aad5a-105">테이블과 같은 원본 개체에 대한 모든 권한을 취소하거나 거부하고 저장 프로시저에 EXECUTE 권한을 부여합니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-105">You revoke or deny all permissions to underlying objects, such as tables, and grant EXECUTE permissions on stored procedures.</span></span> <span data-ttu-id="aad5a-106">그러면 효율적으로 데이터 및 데이터베이스 개체 주변에 보안 경계를 만듭니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-106">This effectively creates a security perimeter around your data and database objects.</span></span>  
  
## <a name="stored-procedure-benefits"></a><span data-ttu-id="aad5a-107">저장 프로시저의 장점</span><span class="sxs-lookup"><span data-stu-id="aad5a-107">Stored Procedure Benefits</span></span>  
 <span data-ttu-id="aad5a-108">저장 프로시저에는 다음과 같은 장점이 있습니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-108">Stored procedures have the following benefits:</span></span>  
  
- <span data-ttu-id="aad5a-109">데이터 논리 및 비즈니스 규칙을 캡슐화하여 개발자 및 데이터베이스 관리자가 의도하는 방식으로만 사용자가 데이터 및 개체에 액세스할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-109">Data logic and business rules can be encapsulated so that users can access data and objects only in ways that developers and database administrators intend.</span></span>  
  
- <span data-ttu-id="aad5a-110">모든 사용자 입력의 유효성을 확인하는 매개 변수화된 저장 프로시저를 사용하여 SQL 삽입 공격을 막을 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-110">Parameterized stored procedures that validate all user input can be used to thwart SQL injection attacks.</span></span> <span data-ttu-id="aad5a-111">동적 SQL을 사용하는 경우 명령을 매개 변수화하고 매개 변수 값을 쿼리 문자열에 직접 넣지 마세요.</span><span class="sxs-lookup"><span data-stu-id="aad5a-111">If you use dynamic SQL, be sure to parameterize your commands, and never include parameter values directly into a query string.</span></span>  
  
- <span data-ttu-id="aad5a-112">임시 쿼리 및 데이터 수정은 허용되지 않습니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-112">Ad hoc queries and data modifications can be disallowed.</span></span> <span data-ttu-id="aad5a-113">실수로 또는 악의적으로 데이터를 파괴하거나 서버 또는 네트워크의 성능을 약화시키는 쿼리를 실행하지 못하도록 합니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-113">This prevents users from maliciously or inadvertently destroying data or executing queries that impair performance on the server or the network.</span></span>  
  
- <span data-ttu-id="aad5a-114">오류를 클라이언트 애플리케이션으로 직접 전달하지 않고 프로시저 코드에서 처리할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-114">Errors can be handled in procedure code without being passed directly to client applications.</span></span> <span data-ttu-id="aad5a-115">그러면 검색 공격을 도울 수 있는 오류 메시지가 반환되지 못합니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-115">This prevents error messages from being returned that could aid in a probing attack.</span></span> <span data-ttu-id="aad5a-116">오류를 기록하고 서버에서 처리합니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-116">Log errors and handle them on the server.</span></span>  
  
- <span data-ttu-id="aad5a-117">저장 프로시저는 한 번만 작성되어 여러 애플리케이션에서 액세스될 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-117">Stored procedures can be written once, and accessed by many applications.</span></span>  
  
- <span data-ttu-id="aad5a-118">클라이언트 애플리케이션에서는 내부 데이터 구조에 대해서는 몰라도 됩니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-118">Client applications do not need to know anything about the underlying data structures.</span></span> <span data-ttu-id="aad5a-119">변경 내용이 매개 변수 목록 또는 반환된 데이터 형식에 영향을 미치지 않는 경우 클라이언트 애플리케이션에서 변경할 필요 없이 저장 프로시저 코드를 변경할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-119">Stored procedure code can be changed without requiring changes in client applications as long as the changes do not affect parameter lists or returned data types.</span></span>  
  
- <span data-ttu-id="aad5a-120">여러 작업을 하나의 프로시저 호출로 결합하므로 저장 프로시저는 네트워크 트래픽을 줄여 줍니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-120">Stored procedures can reduce network traffic by combining multiple operations into one procedure call.</span></span>  
  
## <a name="stored-procedure-execution"></a><span data-ttu-id="aad5a-121">저장 프로시저 실행</span><span class="sxs-lookup"><span data-stu-id="aad5a-121">Stored Procedure Execution</span></span>  
 <span data-ttu-id="aad5a-122">저장 프로시저는 소유권 체인의 장점을 활용하여 데이터에 대한 액세스를 제공하므로 사용자는 데이터베이스 개체에 액세스하기 위해 명시적 권한을 가질 필요가 없습니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-122">Stored procedures take advantage of ownership chaining to provide access to data so that users do not need to have explicit permission to access database objects.</span></span> <span data-ttu-id="aad5a-123">소유권 체인은 순차적으로 서로 액세스하는 개체가 동일한 사용자에 의해 소유될 때 발생합니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-123">An ownership chain exists when objects that access each other sequentially are owned by the same user.</span></span> <span data-ttu-id="aad5a-124">예를 들어 저장 프로시저가 다른 저장 프로시저를 호출할 수 있으며 여러 테이블에 액세스할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-124">For example, a stored procedure can call other stored procedures, or a stored procedure can access multiple tables.</span></span> <span data-ttu-id="aad5a-125">실행 체인에 있는 모든 개체의 소유자가 동일한 경우 SQL Server는 호출자의 다른 개체에 대한 권한이 아닌 호출자에 대한 EXECUTE 권한만 확인합니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-125">If all objects in the chain of execution have the same owner, then SQL Server only checks the EXECUTE permission for the caller, not the caller's permissions on other objects.</span></span> <span data-ttu-id="aad5a-126">따라서 저장 프로시저에 EXECUTE 권한만 부여해야 하며 원본 테이블에 대한 모든 권한을 취소하거나 거부할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-126">Therefore you need to grant only EXECUTE permissions on stored procedures; you can revoke or deny all permissions on the underlying tables.</span></span>  
  
## <a name="best-practices"></a><span data-ttu-id="aad5a-127">모범 사례</span><span class="sxs-lookup"><span data-stu-id="aad5a-127">Best Practices</span></span>  
 <span data-ttu-id="aad5a-128">저장 프로시저를 작성하는 것만으로 애플리케이션이 충분히 보호되는 것은 아닙니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-128">Simply writing stored procedures isn't enough to adequately secure your application.</span></span> <span data-ttu-id="aad5a-129">다음과 같은 잠재적인 보안 허점을 고려해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-129">You should also consider the following potential security holes.</span></span>  
  
- <span data-ttu-id="aad5a-130">저장 프로시저에서 데이터에 액세스하려는 데이터베이스 역할에 대해 EXECUTE 권한을 부여합니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-130">Grant EXECUTE permissions on the stored procedures for database roles you want to be able to access the data.</span></span>  
  
- <span data-ttu-id="aad5a-131">`public` 역할을 포함하여 데이터베이스의 모든 역할과 사용자에 대해 원본 테이블에 대한 모든 권한을 취소하거나 거부합니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-131">Revoke or deny all permissions to the underlying tables for all roles and users in the database, including the `public` role.</span></span> <span data-ttu-id="aad5a-132">모든 사용자는 public에서 권한을 상속합니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-132">All users inherit permissions from public.</span></span> <span data-ttu-id="aad5a-133">따라서 `public`에 대한 권한을 거부하면 소유자 및 `sysadmin` 멤버만 액세스할 수 있으며 다른 모든 사용자는 다른 역할의 멤버에서 권한을 상속할 수 없게 됩니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-133">Therefore denying permissions to `public` means that only owners and `sysadmin` members have access; all other users will be unable to inherit permissions from membership in other roles.</span></span>  
  
- <span data-ttu-id="aad5a-134">`sysadmin` 또는 `db_owner` 역할에 사용자 또는 역할을 추가하지 마세요.</span><span class="sxs-lookup"><span data-stu-id="aad5a-134">Do not add users or roles to the `sysadmin` or `db_owner` roles.</span></span> <span data-ttu-id="aad5a-135">시스템 관리자 및 데이터베이스 소유자가 모든 데이터베이스 개체에 액세스할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-135">System administrators and database owners can access all database objects.</span></span>  
  
- <span data-ttu-id="aad5a-136">`guest` 계정을 비활성화합니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-136">Disable the `guest` account.</span></span> <span data-ttu-id="aad5a-137">그러면 익명 사용자가 데이터베이스에 연결하지 못합니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-137">This will prevent anonymous users from connecting to the database.</span></span> <span data-ttu-id="aad5a-138">새 데이터베이스에서 guest 계정은 기본적으로 비활성화됩니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-138">The guest account is disabled by default in new databases.</span></span>  
  
- <span data-ttu-id="aad5a-139">오류 처리를 구현하고 오류를 기록합니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-139">Implement error handling and log errors.</span></span>  
  
- <span data-ttu-id="aad5a-140">모든 사용자 입력의 유효성을 확인하는 매개 변수화된 저장 프로시저를 만듭니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-140">Create parameterized stored procedures that validate all user input.</span></span> <span data-ttu-id="aad5a-141">모든 사용자 입력을 신뢰되지 않은 것으로 처리합니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-141">Treat all user input as untrusted.</span></span>  
  
- <span data-ttu-id="aad5a-142">동적 SQL은 반드시 필요한 경우에만 사용하세요.</span><span class="sxs-lookup"><span data-stu-id="aad5a-142">Avoid dynamic SQL unless absolutely necessary.</span></span> <span data-ttu-id="aad5a-143">입력 문자열에서 문자열 값을 구분하고 구분 기호가 생기지 않도록 Transact-SQL QUOTENAME() 함수를 사용합니다.</span><span class="sxs-lookup"><span data-stu-id="aad5a-143">Use the Transact-SQL QUOTENAME() function to delimit a string value and escape any occurrence of the delimiter in the input string.</span></span>
