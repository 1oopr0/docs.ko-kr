- <span data-ttu-id="3104e-241">상속 계층 구조에서는 모든 메시지 헤더가 함께 수집되어 메시지에 대한 전체 헤더 집합을 형성합니다.</span><span class="sxs-lookup"><span data-stu-id="3104e-241">All of the message headers in the inheritance hierarchy are collected together to form the full set of headers for the message.</span></span>

  [MessageHeader(Name="ID")] public int personID;  
  [MessageBodyMember] public string patientName;  
}  
  
[MessageContract]

  [MessageHeader(Name="ID")] public int patientID;

}

 <span data-ttu-id="3104e-247">`PatientRecord` 클래스는 `ID`라는 하나의 헤더로 메시지를 설명합니다.</span><span class="sxs-lookup"><span data-stu-id="3104e-247">The `PatientRecord` class describes a message with one header called `ID`.</span></span> <span data-ttu-id="3104e-248">기본-최상 멤버를 선택했으므로 헤더는 `personID` 멤버가 아닌 `patientID`에 해당합니다.</span><span class="sxs-lookup"><span data-stu-id="3104e-248">The header corresponds to the `personID` and not the `patientID` member, because the base-most member is chosen.</span></span> <span data-ttu-id="3104e-249">따라서 이 경우에는 `patientID` 필드가 필요하지 않습니다.</span><span class="sxs-lookup"><span data-stu-id="3104e-249">Thus, the `patientID` field is useless in this case.</span></span> <span data-ttu-id="3104e-250">사전순이므로 메시지 본문에는 `diagnosis` 요소가 `patientName` 요소보다 먼저 포함됩니다.</span><span class="sxs-lookup"><span data-stu-id="3104e-250">The body of the message contains the `diagnosis` element followed by the `patientName` element, because that is the alphabetical order.</span></span> <span data-ttu-id="3104e-251">예제에서는 사용해서는 안 되는 패턴을 보여 줍니다. 기본 메시지 계약 및 파생 메시지 계약에는 모두 메시지 본문 부분이 있습니다.</span><span class="sxs-lookup"><span data-stu-id="3104e-251">Notice that the example shows a pattern that is strongly discouraged: both the base and the derived message contracts have message body parts.</span></span>  
  
## <a name="wsdl-considerations"></a><span data-ttu-id="3104e-252">WSDL 고려 사항</span><span class="sxs-lookup"><span data-stu-id="3104e-252">WSDL Considerations</span></span>

## <a name="soap-encoding-considerations"></a><span data-ttu-id="3104e-262">SOAP 인코딩 고려 사항</span><span class="sxs-lookup"><span data-stu-id="3104e-262">SOAP Encoding Considerations</span></span>

## <a name="performance-considerations"></a><span data-ttu-id="3104e-274">성능 고려 사항</span><span class="sxs-lookup"><span data-stu-id="3104e-274">Performance Considerations</span></span>

 <span data-ttu-id="3104e-283">메시지 개체를 `Result` 속성으로 수신하고 반환된 값을 해당 개체의 속성으로 포함하려면 `/messageContract` 명령 옵션을 사용합니다.</span><span class="sxs-lookup"><span data-stu-id="3104e-283">If you want to receive the message object as the `Result` property and have the returned values as properties on that object, use the `/messageContract` command option.</span></span> <span data-ttu-id="3104e-284">이렇게 하면 응답 메시지를 `Result` 개체의 <xref:System.EventArgs> 속성으로 반환하는 서명이 생성됩니다.</span><span class="sxs-lookup"><span data-stu-id="3104e-284">This generates a signature that returns the response message as the `Result` property on the <xref:System.EventArgs> object.</span></span> <span data-ttu-id="3104e-285">모든 내부 반환 값은 응답 메시지 개체의 속성이 됩니다.</span><span class="sxs-lookup"><span data-stu-id="3104e-285">All internal return values are then properties of the response message object.</span></span>
