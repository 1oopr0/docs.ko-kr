## <a name="how-an-object-ends-up-on-the-loh"></a><span data-ttu-id="4cbcd-111">개체가 LOH에 배치되는 방식</span><span class="sxs-lookup"><span data-stu-id="4cbcd-111">How an object ends up on the LOH</span></span>

<span data-ttu-id="4cbcd-112">개체 크기가 85,000바이트 이상이면 큰 개체로 간주됩니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-112">If an object is greater than or equal to 85,000 bytes in size, it’s considered a large object.</span></span> <span data-ttu-id="4cbcd-113">이 숫자는 성능 튜닝으로 결정됩니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-113">This number was determined by performance tuning.</span></span> <span data-ttu-id="4cbcd-114">개체 할당 요청이 85,000바이트 이상이면 런타임에서 이를 큰 개체 힙에 할당합니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-114">When an object allocation request is for 85,000 or more bytes, the runtime allocates it on the large object heap.</span></span>

<span data-ttu-id="4cbcd-115">이 의미를 이해하려면 가비지 수집기에 대한 몇 가지 기본 사항을 검토하는 것이 유용합니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-115">To understand what this means, it's useful to examine some fundamentals about the garbage collector.</span></span>

<span data-ttu-id="4cbcd-116">가비지 수집기는 세대별 수집기입니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-116">The garbage collector is a generational collector.</span></span> <span data-ttu-id="4cbcd-117">세 개의 세대, 즉 0세대, 1세대 및 2세대가 있습니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-117">It has three generations: generation 0, generation 1, and generation 2.</span></span> <span data-ttu-id="4cbcd-118">세 개의 세대가 있는 이유는 잘 조정된 응용 프로그램에서 대부분의 개체가 0세대에서 소멸된다는 것입니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-118">The reason for having 3 generations is that, in a well-tuned app, most objects die in gen0.</span></span> <span data-ttu-id="4cbcd-119">예를 들어 서버 응용 프로그램에서는 요청이 완료되면 각 요청과 관련된 할당이 소멸되어야 합니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-119">For example, in a server app, the allocations associated with each request should die after the request is finished.</span></span> <span data-ttu-id="4cbcd-120">진행 중인 할당 요청은 1세대가 되고 여기서 소멸됩니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-120">The in-flight allocation requests will make it into gen1 and die there.</span></span> <span data-ttu-id="4cbcd-121">기본적으로 1세대는 최근에 생성된 개체 영역과 수명이 긴 개체 영역 간의 버퍼 역할을 합니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-121">Essentially, gen1 acts as a buffer between young object areas and long-lived object areas.</span></span>

<span data-ttu-id="4cbcd-122">작은 개체는 항상 0세대에 할당되며 수명에 따라 1세대 또는 2세대로 승격될 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-122">Small objects are always allocated in generation 0 and, depending on their lifetime, may be promoted to generation 1 or generation2.</span></span> <span data-ttu-id="4cbcd-123">큰 개체는 항상 2세대에 할당됩니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-123">Large objects are always allocated in generation 2.</span></span>

<span data-ttu-id="4cbcd-124">큰 개체는 2세대 수집 동안에만 수집되므로 2세대에 속합니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-124">Large objects belong to generation 2 because they are collected only during a generation 2 collection.</span></span> <span data-ttu-id="4cbcd-125">특정 세대가 수집되면 그 이전 세대도 모두 수집됩니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-125">When a generation is collected, all its younger generation(s) are also collected.</span></span> <span data-ttu-id="4cbcd-126">예를 들어 1세대 GC가 수행되면 0세대와 1세대가 모두 수집되고,</span><span class="sxs-lookup"><span data-stu-id="4cbcd-126">For example, when a generation 1 GC happens, both generation 1 and 0 are collected.</span></span> <span data-ttu-id="4cbcd-127">2세대 GC가 수행되면 전체 힙이 수집됩니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-127">And when a generation 2 GC happens, the whole heap is collected.</span></span> <span data-ttu-id="4cbcd-128">이러한 이유로 2세대 GC는 *전체 GC*라고도 합니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-128">For this reason, a generation 2 GC is also called a *full GC*.</span></span> <span data-ttu-id="4cbcd-129">이 문서에서는 전체 GC 대신 2세대 GC를 언급하지만 용어는 서로 바꿔 사용할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-129">This article refers to generation 2 GC instead of full GC, but the terms are interchangeable.</span></span>

<span data-ttu-id="4cbcd-130">세대는 GC 힙의 논리적 뷰를 제공합니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-130">Generations provide a logical view of the GC heap.</span></span> <span data-ttu-id="4cbcd-131">실제로 개체는 관리되는 힙 세그먼트에 있습니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-131">Physically, objects live in managed heap segments.</span></span> <span data-ttu-id="4cbcd-132">*관리되는 힙 세그먼트*는 GC에서 관리 코드 대신 [VirtualAlloc 함수](/windows/desktop/api/memoryapi/nf-memoryapi-virtualalloc)를 호출하여 OS로부터 예약하는 메모리 청크입니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-132">A *managed heap segment* is a chunk of memory that the GC reserves from the OS by calling the [VirtualAlloc function](/windows/desktop/api/memoryapi/nf-memoryapi-virtualalloc) on behalf of managed code.</span></span> <span data-ttu-id="4cbcd-133">CLR이 로드되면 GC에서 작은 개체(SOH 또는 작은 개체 힙)와 큰 개체(큰 개체 힙)에 대해 하나씩 두 개의 초기 힙 세그먼트를 할당합니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-133">When the CLR is loaded, the GC allocates two initial heap segments: one for small objects (the small object heap, or SOH), and one for large objects (the large object heap).</span></span>

<span data-ttu-id="4cbcd-134">그런 다음, 이러한 관리되는 힙 세그먼트에 관리되는 개체를 배치하여 할당 요청이 충족됩니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-134">The allocation requests are then satisfied by putting managed objects on these managed heap segments.</span></span> <span data-ttu-id="4cbcd-135">개체가 85,000바이트보다 작으면 SOH용 세그먼트에 배치되며, 그렇지 않으면 LOH 세그먼트에 배치됩니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-135">If the object is less than 85,000 bytes, it is put on the segment for the SOH; otherwise, it is put on an LOH segment.</span></span> <span data-ttu-id="4cbcd-136">세그먼트에 더 많은 개체가 할당됨에 따라 세그먼트가 더 작은 청크로 커밋됩니다.</span><span class="sxs-lookup"><span data-stu-id="4cbcd-136">Segments are committed (in smaller chunks) as more and more objects are allocated onto them.</span></span>
